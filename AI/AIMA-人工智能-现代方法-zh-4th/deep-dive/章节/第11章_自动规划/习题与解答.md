### 习题11.1 【难度：⭐】PDDL动作模式理解

**知识点**：PDDL语法、前提条件、效果

**题目**：
考虑以下PDDL动作模式：
```
Action(Fly(p, from, to),
  PRECOND: At(p, from) ∧ Plane(p) ∧ Airport(from) ∧ Airport(to)
  EFFECT: ¬At(p, from) ∧ At(p, to))
```

请回答：
1. 该动作模式定义了什么类型的动作？
2. 如果要执行Fly(P₁, SFO, JFK)，需要满足哪些前提条件？
3. 执行该动作后，状态会发生什么变化？

**逐步提示**：
- 注意动作名称Fly表明这是飞行动作
- 前提条件中的∧表示所有条件必须同时满足
- 效果中的¬表示删除列表（delete list）

**详细解答**：

1. **动作类型**：这是一个飞行动作模式，定义了飞机p从机场from飞往机场to的操作。

2. **前提条件**：
   - At(P₁, SFO)：飞机P₁必须在SFO机场
   - Plane(P₁)：P₁必须是一架飞机
   - Airport(SFO)：SFO必须是机场
   - Airport(JFK)：JFK必须是机场

3. **状态变化**：
   - 删除（Delete）：At(P₁, SFO) 从状态中移除
   - 添加（Add）：At(P₁, JFK) 添加到状态中
   
   用公式表示：Result(s, a) = (s - DEL(a)) ∪ ADD(a)

**考点解析**：
- 理解PDDL动作模式的三个组成部分：动作名称、PRECOND（前提）、EFFECT（效果）
- 掌握添加列表（ADD）和删除列表（DEL）的概念
- 理解封闭世界假设：未提及的流默认为假

### 习题11.2 【难度：⭐】积木世界规划

**知识点**：积木世界、Clear谓词、Move动作

**题目**：
在积木世界中，初始状态为：
- On(A, Table), On(B, Table), On(C, A)
- Clear(B), Clear(C), Clear(Table)

目标状态为：On(A, B) ∧ On(B, C)

动作定义：
```
Action(Move(b, x, y),
  PRECOND: On(b, x) ∧ Clear(b) ∧ Clear(y) ∧ Block(b) ∧ Block(y) ∧ (b≠x) ∧ (b≠y) ∧ (x≠y)
  EFFECT: On(b, y) ∧ Clear(x) ∧ ¬On(b, x) ∧ ¬Clear(y))
```

请给出从初始状态到目标状态的一个有效规划（动作序列）。

**逐步提示**：
- 首先分析初始状态和目标状态的差异
- 注意C在A上，但目标中B在C上
- 需要先移动C，才能移动A到B上

**详细解答**：

**初始状态分析**：
- 积木C在A上，B在桌上，A在桌上
- 目标：A在B上，B在C上（形成一个塔）

**规划步骤**：

1. **Move(C, A, Table)**：将C从A移到桌上
   - 前提满足：On(C,A)✓, Clear(C)✓, Clear(Table)✓
   - 效果：On(C,Table), Clear(A), ¬On(C,A), ¬Clear(Table)

2. **Move(B, Table, C)**：将B移到C上
   - 前提满足：On(B,Table)✓, Clear(B)✓, Clear(C)✓
   - 效果：On(B,C), Clear(Table), ¬On(B,Table), ¬Clear(C)

3. **Move(A, Table, B)**：将A移到B上
   - 前提满足：On(A,Table)✓, Clear(A)✓, Clear(B)✓
   - 效果：On(A,B), Clear(Table), ¬On(A,Table), ¬Clear(B)

**最终规划**：[Move(C, A, Table), Move(B, Table, C), Move(A, Table, B)]

**考点解析**：
- 积木世界是经典的规划领域
- Clear谓词表示上方无积木
- 理解动作的前提条件如何限制规划的合法性
- 规划需要满足子目标之间的顺序约束

### 习题11.3 【难度：⭐⭐】航空货物运输规划

**知识点**：PDDL建模、多货物运输

**题目**：
考虑以下航空货物运输问题：
- 货物：C₁在SFO，C₂在JFK
- 飞机：P₁在SFO，P₂在JFK
- 目标：C₁在JFK，C₂在SFO

动作定义：
```
Action(Load(c, p, a),
  PRECOND: At(c,a) ∧ At(p,a) ∧ Cargo(c) ∧ Plane(p) ∧ Airport(a)
  EFFECT: ¬At(c,a) ∧ In(c,p))

Action(Unload(c, p, a),
  PRECOND: In(c,p) ∧ At(p,a) ∧ Cargo(c) ∧ Plane(p) ∧ Airport(a)
  EFFECT: At(c,a) ∧ ¬In(c,p))

Action(Fly(p, from, to),
  PRECOND: At(p,from) ∧ Plane(p) ∧ Airport(from) ∧ Airport(to)
  EFFECT: ¬At(p,from) ∧ At(p,to))
```

请给出一个完整的规划将C₁运到JFK、C₂运到SFO。

**逐步提示**：
- 每架飞机只能运一件货物
- 货物必须在飞机所在位置才能装载
- 注意飞机和货物的初始位置

**详细解答**：

**规划步骤**：

1. **Load(C₁, P₁, SFO)**：将C₁装入P₁
   - 前提：C₁在SFO，P₁在SFO ✓

2. **Fly(P₁, SFO, JFK)**：P₁飞往JFK
   - 前提：P₁在SFO ✓

3. **Unload(C₁, P₁, JFK)**：在JFK卸载C₁
   - 前提：C₁在P₁中，P₁在JFK ✓

4. **Load(C₂, P₂, JFK)**：将C₂装入P₂
   - 前提：C₂在JFK，P₂在JFK ✓

5. **Fly(P₂, JFK, SFO)**：P₂飞往SFO
   - 前提：P₂在JFK ✓

6. **Unload(C₂, P₂, SFO)**：在SFO卸载C₂
   - 前提：C₂在P₂中，P₂在SFO ✓

**最终规划**：[Load(C₁, P₁, SFO), Fly(P₁, SFO, JFK), Unload(C₁, P₁, JFK), Load(C₂, P₂, JFK), Fly(P₂, JFK, SFO), Unload(C₂, P₂, SFO)]

**考点解析**：
- 理解In谓词表示货物在飞机内部
- 货物随飞机移动（通过In关系间接表示）
- 规划问题可能包含独立子问题

### 习题11.4 【难度：⭐⭐】前向搜索与反向搜索

**知识点**：前向状态空间搜索、反向状态空间搜索

**题目**：
考虑备用轮胎问题：
- 初始：Tire(Flat), Tire(Spare), At(Flat, Axle), At(Spare, Trunk)
- 目标：At(Spare, Axle)
- 动作：Remove(obj, loc), PutOn(t, Axle), LeaveOvernight

**问题**：
1. 在前向搜索中，从初始状态可以执行哪些动作？为什么？
2. 在反向搜索中，从目标状态可以反向应用哪些相关动作？为什么？
3. 哪种搜索方式在这个问题中分支因子更小？为什么？

**逐步提示**：
- 前向搜索考虑所有前提被满足的动作
- 反向搜索考虑效果与目标中文字合一的动作
- 比较两种搜索的候选动作数量

**详细解答**：

1. **前向搜索的适用动作**：
   - **Remove(Flat, Axle)**：前提At(Flat, Axle) ✓
   - **Remove(Spare, Trunk)**：前提At(Spare, Trunk) ✓
   - **LeaveOvernight**：无前提，总是适用
   
   共3个动作可选。

2. **反向搜索的相关动作**：
   目标文字：At(Spare, Axle)
   
   - **PutOn(Spare, Axle)**：效果包含At(Spare, Axle)，且不否定目标其他部分 ✓
   - **LeaveOvernight**：效果包含¬At(Spare, Axle)，否定目标 ✗
   
   只有PutOn(Spare, Axle)是相关的。

3. **分支因子比较**：
   - 前向搜索分支因子：3（Remove(Flat,Axle)、Remove(Spare,Trunk)、LeaveOvernight）
   - 反向搜索分支因子：1（仅PutOn）
   
   **反向搜索分支因子更小**。反向搜索只考虑能够实现目标文字的动作，而前向搜索必须考虑所有当前可执行的动作。

**考点解析**：
- 前向搜索：从初始状态出发，应用所有适用动作
- 反向搜索：从目标出发，反向应用相关动作
- 相关动作定义：效果能与目标文字合一，但不否定目标其他部分
- 反向搜索通常分支因子更小，但难以找到好的启发式

### 习题11.5 【难度：⭐⭐】忽略删除列表启发式

**知识点**：松弛问题、启发式函数

**题目**：
在积木世界中，忽略删除列表启发式从所有动作中移除删除列表中的负文字。

原始Move动作：
```
Action(Move(b, x, y),
  PRECOND: On(b, x) ∧ Clear(b) ∧ Clear(y)
  EFFECT: On(b, y) ∧ Clear(x) ∧ ¬On(b, x) ∧ ¬Clear(y))
```

**问题**：
1. 移除删除列表后，Move动作的修改效果是什么？
2. 这种松弛如何简化规划问题？
3. 为什么这种启发式是可采纳的（不会高估真实代价）？

**逐步提示**：
- 删除列表包含¬On(b, x)和¬Clear(y)
- 考虑移除这些负效果后规划的性质
- 比较松弛问题与原问题的解

**详细解答**：

1. **修改后的效果**：
   移除删除列表后，效果变为：On(b, y) ∧ Clear(x)
   
   即：只添加On(b, y)和Clear(x)，不删除任何内容。

2. **松弛问题的简化**：
   - 积木一旦在某位置，就永远在那里（无法删除On关系）
   - Clear一旦为真，就永远为真
   - 规划变成单调积累目标条件的过程
   - 不会出现动作相互抵消（undo）的情况
   - 可以通过简单的爬山搜索求解

3. **可采纳性原因**：
   - 松弛问题比原问题更容易求解（约束更少）
   - 原问题的任何解都是松弛问题的解
   - 松弛问题可能有更短的解
   - 因此松弛问题的解长度 ≤ 原问题的解长度
   - 启发式值不会高估真实代价

**考点解析**：
- 忽略删除列表是经典规划中最强大的领域无关启发式之一
- 松弛问题通过移除负效果使规划单调前进
- 可采纳启发式保证A*搜索的最优性

### 习题11.6 【难度：⭐⭐⭐】启发式规划算法

**知识点**：忽略前提启发式、规划图、FF规划器

**题目**：
考虑航空货物运输问题，有10个机场、50架飞机、200件货物。目标是将所有在A机场的货物运送到B机场。

**问题**：
1. 使用前向搜索时，为什么需要好的启发式函数？
2. 解释忽略前提启发式如何估计距离目标的状态距离。
3. 解释忽略删除列表启发式为什么比忽略前提启发式更精确。

**逐步提示**：
- 计算前向搜索的分支因子
- 分析两种松弛方法的差异
- 比较启发式的信息量和计算复杂度

**详细解答**：

1. **需要好启发式的原因**：
   - 分支因子极大：每架飞机可飞到9个其他机场，每件货物可装卸
   - 最少有约450种动作，最多约10,450种动作
   - 解长度约41步（20件×2 + 1飞行）
   - 搜索空间约为 2000⁴¹，没有启发式无法求解

2. **忽略前提启发式**：
   - 从所有动作中移除前提
   - 任何动作都可在任何状态执行
   - 启发式值 = 未满足的目标数量
   - 每个目标文字可在一步内完成
   - 对于200件货物，h = 200（严重低估）

3. **忽略删除列表启发式更精确**：
   - 保留动作前提，只移除删除列表
   - 考虑动作之间的依赖关系
   - 可以通过图规划算法（Planning Graph）计算更精确的估计
   - 考虑多个目标可由一个动作同时满足的情况
   - 对于货物运输，考虑飞机容量约束

**考点解析**：
- FF规划器结合忽略删除列表启发式和规划图估计
- 启发式的质量直接影响搜索效率
- 需要在启发式计算开销和精度之间权衡

### 习题11.7 【难度：⭐⭐⭐】分层任务网络(HTN)

**知识点**：高层动作(HLA)、细化、分层规划

**题目**：
考虑以下HTN规划问题：

高层动作：
```
Refinement(Go(Home, SFO),
  STEPS: [Drive(Home, SFOLongTermParking), Shuttle(SFOLongTermParking, SFO)])

Refinement(Go(Home, SFO),
  STEPS: [Taxi(Home, SFO)])
```

基元动作：
- Drive(x, y)：从x开车到y
- Shuttle(x, y)：从x乘摆渡车到y
- Taxi(x, y)：从x乘出租车到y

**问题**：
1. 高层动作Go(Home, SFO)有多少种实现？
2. 如果一个高层规划包含两个独立的Go动作：Go(Home, SFO)和Go(SFO, Hotel)，这个高层规划的实现数量是多少？
3. 解释分层搜索如何降低计算复杂度。

**逐步提示**：
- 每个高层动作可能有多个细化
- 独立子问题的解可以组合
- 比较分层搜索与非分层搜索的复杂度

**详细解答**：

1. **Go(Home, SFO)的实现数量**：
   - 细化1：[Drive(Home, SFOLongTermParking), Shuttle(SFOLongTermParking, SFO)]
   - 细化2：[Taxi(Home, SFO)]
   - 共**2种实现**

2. **两个Go动作的实现数量**：
   - 第一个Go(Home, SFO)：2种选择
   - 第二个Go(SFO, Hotel)：假设也有2种选择
   - 总实现数量 = 2 × 2 = **4种**
   
   高层规划：[Go(Home, SFO), Go(SFO, Hotel)]
   
   可能的基元实现：
   - [Drive, Shuttle, Drive, Shuttle]
   - [Drive, Shuttle, Taxi]
   - [Taxi, Drive, Shuttle]
   - [Taxi, Taxi]

3. **分层搜索的复杂度降低**：
   
   设：
   - 解深度d = 10个基元动作
   - 每个HLA有r = 2个细化
   - 每个细化产生k = 3个基元动作
   
   **非分层搜索**：O(bᵈ) = O(b¹⁰)
   
   **分层搜索**：
   - 内部细化节点数：(d-1)/(k-1) = (10-1)/(3-1) = 4.5 ≈ 5
   - 可能的分解树数量：r^((d-1)/(k-1)) = 2^5 = 32
   
   复杂度从指数级降低到关于解深度的线性或多项式级。

**考点解析**：
- HTN利用领域知识通过高层动作分解问题
- 分层搜索在高层规划空间搜索，而非基元动作空间
- 复杂度降低取决于r（每个HLA的细化数）和k（每个细化产生的动作数）
- r小、k大的情况下，分层搜索优势最明显

### 习题11.8 【难度：⭐⭐⭐】非确定性域的规划

**知识点**：无传感器规划、条件效果、信念状态

**题目**：
考虑刷漆问题的简化版本：
- 初始：物体A和B未知颜色，油漆罐Can1和Can2未知颜色
- 目标：A和B颜色相同

动作：
```
Action(Paint(x, can),
  PRECOND: Object(x) ∧ Can(can) ∧ Color(can, c) ∧ Open(can)
  EFFECT: Color(x, c))
```

**问题**：
1. 无传感器智能体如何求解此问题（不使用传感器）？
2. 如果Paint动作有条件效果（Color(can, c)为前提的一部分，但未在动作变量中声明），这在PDDL中如何表示？
3. 解释为什么条件效果会导致信念状态不在1-CNF中。

**逐步提示**：
- 考虑如何通过动作强制实现目标
- 分析条件效果对信念状态表示的影响
- 理解1-CNF信念状态的闭包性质

**详细解答**：

1. **无传感器智能体的解法**：
   - 打开Can1（RemoveLid(Can1)）
   - 用Can1给A刷漆（Paint(A, Can1)）
   - 用Can1给B刷漆（Paint(B, Can1)）
   
   无论Can1中是什么颜色，A和B都会被刷成相同颜色。智能体不需要知道具体颜色，只需要执行相同的动作序列。

2. **PDDL表示**：
   ```
   Action(Paint(x, can),
     PRECOND: Object(x) ∧ Can(can) ∧ Open(can)
     EFFECT: ∃c Color(can, c) ∧ Color(x, c))
   ```
   
   或者使用带变量的效果（PDDL扩展）：
   ```
   Action(Paint(x, can),
     PRECOND: Object(x) ∧ Can(can) ∧ Open(can) ∧ Color(can, c)
     EFFECT: Color(x, c))
   ```
   
   这里c不是动作参数，而是效果的变量。

3. **条件效果与1-CNF**：
   - 初始信念状态：Color(x, C(x))（所有物体有某种未知颜色）
   - 执行Paint(A, Can1)后：Color(A, C(Can1))
   - 现在A的颜色与Can1的颜色相关联
   - 如果执行Paint(B, Can2)且Can2颜色未知：Color(B, C(Can2))
   - 如果Can1和Can2可能是同一罐油漆，则Color(A, c)和Color(B, c)之间有依赖关系
   - 这种依赖关系无法用简单的文字合取表示
   - 信念状态需要表示为：Color(A, c₁) ∧ Color(B, c₂) ∧ (c₁ = c₂ ∨ c₁ ≠ c₂)
   - 这不再是1-CNF形式

**考点解析**：
- 无传感器规划通过动作设计强制实现目标
- 条件效果引入变量间的依赖关系
- 1-CNF信念状态在无条件动作下是闭的，但在条件效果下不是
- 这是规划与概率推理交叉的核心问题

### 习题11.9 【难度：⭐⭐⭐】在线规划与重规划

**知识点**：执行监控、动作监控、规划监控、重规划

**题目**：
考虑油漆问题的应变规划执行：
```
[LookAt(Table), LookAt(Chair),
 if Color(Table,c) ∧ Color(Chair,c) then NoOp
 else [RemoveLid(Can1), LookAt(Can1),
       if Color(Table,c) ∧ Color(Can1,c) then Paint(Chair,Can1)
       else [Paint(Chair,Can1), Paint(Table,Can1)]]]
```

**问题**：
1. 解释动作监控和规划监控的区别。
2. 如果使用动作监控，当Paint(Chair, Can1)失败（椅子仍是黑色）时会发生什么？
3. 如果使用规划监控，上述情况会有什么不同？
4. 为什么规划监控通常比动作监控更智能？

**逐步提示**：
- 动作监控只检查当前动作的前提
- 规划监控检查剩余规划的整个成功可能性
- 比较两种监控方式的响应行为

**详细解答**：

1. **动作监控 vs 规划监控**：
   
   **动作监控**：
   - 在执行每个动作前，验证该动作的前提是否成立
   - 如果不成立，触发重规划
   - 只关注当前动作
   
   **规划监控**：
   - 在执行动作前，验证剩余规划是否仍然可以成功
   - 检查规划中所有步骤的前提（除了那些会被剩余规划中某步骤达成的）
   - 关注整个剩余规划的可行性

2. **动作监控的行为**：
   - Paint(Chair, Can1)执行前检查前提（Object(Chair), Can(Can1), Open(Can1), Color(Can1,c)）
   - 前提都满足，执行动作
   - 动作后检查：发现椅子仍是黑色（与预期效果Color(Chair,c)不符）
   - 检测到失败，触发重规划
   - 可能重试Paint(Chair, Can1)或尝试其他方案

3. **规划监控的行为**：
   - 在执行Paint(Chair, Can1)前，检查剩余规划是否能成功
   - 剩余规划要求最终Color(Chair,c) ∧ Color(Table,c)
   - 如果预测Paint可能失败（如油漆不足），会提前检测到问题
   - 在执行Paint前就触发重规划

4. **规划监控更智能的原因**：
   - **提前检测**：能在执行前发现规划无法成功，避免浪费时间
   - **更智能的响应**：可以评估不同修复方案的代价
   - **允许意外收获**：如果偶然达到目标，可以提前结束
   - 例子：如果油漆只够刷椅子，动作监控会刷完椅子才发现无法刷桌子；规划监控会在刷椅子前就发现整个规划不可行。

**考点解析**：
- 执行监控是健壮智能体的关键能力
- 规划监控考虑前瞻性信息，通常优于动作监控
- 实际系统中常结合多种监控方式

### 习题11.10 【难度：⭐⭐⭐⭐】天使语义与可达集

**知识点**：天使非确定性、可达集、近似描述

**题目**：
考虑以下两个高层动作：

```
Action(h₁, PRECOND: ¬A, EFFECT: A ∧ ~B)  // ~B 表示"可能删除B"
Action(h₂, PRECOND: ¬B, EFFECT: ~A ∧ ±C)  // ~A 表示"可能添加A"，±C表示"可能添加或删除C"
```

**问题**：
1. 解释~B和±C的天使语义含义。
2. 从初始状态{A: false, B: true, C: false}，高层规划[h₁, h₂]的乐观可达集是什么？
3. 悲观可达集是什么？
4. 给定目标A ∧ C，判断这个规划是否可行。

**逐步提示**：
- 天使语义：智能体选择实现方式
- 乐观描述：可能夸大的可达集
- 悲观描述：可能低估的可达集
- 判断可达集与目标的交集

**详细解答**：

1. **天使语义含义**：
   
   **~B（可能删除B）**：
   - 如果B为真，可以选择删除B或保持B
   - 如果B为假，B保持为假
   - 即：B可以被智能体控制删除
   
   **±C（可能添加或删除C）**：
   - 如果C为真，可以选择删除C或保持C
   - 如果C为假，可以选择添加C或保持C
   - 即：C可以被智能体完全控制（设为真或假）

2. **乐观可达集 REACH⁺(s, [h₁, h₂])**：
   
   初始状态：{A=F, B=T, C=F}
   
   执行h₁后可能状态：
   - 选择实现h₁的方式使B=T：{A=T, B=T, C=F}
   - 选择实现h₁的方式使B=F：{A=T, B=F, C=F}
   
   从{A=T, B=T, C=F}执行h₂：
   - 前提¬B不满足，h₂不可执行
   
   从{A=T, B=F, C=F}执行h₂：
   - 选择±C使C=T：{A=T, B=F, C=T}
   - 选择±C使C=F：{A=T, B=F, C=F}
   - 选择~A保持A=T：{A=T, B=F, ...}
   
   **乐观可达集**：{A=T, B=F, C=T} 和 {A=T, B=F, C=F}

3. **悲观可达集 REACH⁻(s, [h₁, h₂])**：
   
   悲观描述只包含所有实现都能达到的效果。
   
   对于h₁：
   - 所有实现都添加A
   - 不是所有实现都删除B（~B表示"可能"而非"必然"）
   - 悲观效果：{A=T}
   
   对于h₂：
   - 从悲观状态{A=T, B=F, C=F}执行
   - ~A不是所有实现都添加A（实际上¬B的前提满足，但~A表示"可能添加A"）
   - 如果A已为真，~A可以保持不变
   - ±C不是所有实现都设C为特定值
   
   **悲观可达集**：{A=T, B=F, C=F}（不确定C能否为真）

4. **可行性判断**：
   
   目标：A ∧ C
   
   - 乐观可达集 REACH⁺ 包含 {A=T, B=F, C=T}，与目标相交 ✓
   - 悲观可达集 REACH⁻ 包含 {A=T, B=F, C=F}，与目标不相交 ✗
   
   结论：**规划可能可行但不确定**。需要进一步细化h₂的实现，特别是确认是否能控制C为真。

**考点解析**：
- 天使语义与恶魔语义的区别（智能体选择 vs 对手选择）
- 可达集计算是分层规划的核心
- 乐观和悲观描述用于高效判断规划可行性
- 近似描述导致"不确定"区域，需要进一步细化

### 习题11.11 【难度：⭐⭐⭐⭐】时间调度与关键路径

**知识点**：关键路径方法(CPM)、松弛时间、资源约束

**题目**：
考虑以下作业车间调度问题：

| 动作 | 持续时间 | 前置动作 | 所需资源 |
|------|---------|---------|---------|
| A: AddEngine₁ | 30 | - | EngineHoist(1) |
| B: AddEngine₂ | 60 | - | EngineHoist(1) |
| C: AddWheels₁ | 30 | A | WheelStation(1), LugNuts(20) |
| D: AddWheels₂ | 15 | B | WheelStation(1), LugNuts(20) |
| E: Inspect₁ | 10 | C | Inspectors(1) |
| F: Inspect₂ | 10 | D | Inspectors(1) |

资源：EngineHoist=1, WheelStation=1, Inspectors=2, LugNuts=500

**问题**：
1. 不考虑资源约束，计算每个动作的最早开始时间(ES)和最晚开始时间(LS)。
2. 找出关键路径。
3. 考虑资源约束（EngineHoist=1），新的调度是什么？
4. 如果Inspectors只有1人，调度会如何变化？

**逐步提示**：
- 使用动态规划计算ES和LS
- 关键路径是ES=LS的动作序列
- 资源约束需要解决冲突（两个动作不能同时执行）

**详细解答**：

1. **计算ES和LS**：
   
   **正向计算ES**：
   - ES(A) = 0, EF(A) = 30
   - ES(B) = 0, EF(B) = 60
   - ES(C) = EF(A) = 30, EF(C) = 60
   - ES(D) = EF(B) = 60, EF(D) = 75
   - ES(E) = EF(C) = 60, EF(E) = 70
   - ES(F) = EF(D) = 75, EF(F) = 85
   
   项目完成时间 = max(70, 85) = 85
   
   **反向计算LS**：
   - LF(F) = 85, LS(F) = 75
   - LF(E) = 85, LS(E) = 75
   - LF(D) = LS(F) = 75, LS(D) = 60
   - LF(C) = LS(E) = 75, LS(C) = 45
   - LF(B) = LS(D) = 60, LS(B) = 0
   - LF(A) = LS(C) = 45, LS(A) = 15
   
   | 动作 | 持续时间 | ES | EF | LS | LF | 松弛(LS-ES) |
   |-----|---------|----|----|----|----|------------|
   | A | 30 | 0 | 30 | 15 | 45 | 15 |
   | B | 60 | 0 | 60 | 0 | 60 | 0 |
   | C | 30 | 30 | 60 | 45 | 75 | 15 |
   | D | 15 | 60 | 75 | 60 | 75 | 0 |
   | E | 10 | 60 | 70 | 75 | 85 | 15 |
   | F | 10 | 75 | 85 | 75 | 85 | 0 |

2. **关键路径**：
   关键路径上的动作松弛为0：
   **B → D → F**
   
   项目持续时间：60 + 15 + 10 = 85

3. **考虑EngineHoist=1约束**：
   
   A和B都需要EngineHoist，不能同时执行。
   
   两种可能：
   - A先：A[0-30], B[30-90], 然后C[30-60], D[90-105], E[60-70], F[105-115]
     总时间 = 115
   - B先：B[0-60], A[60-90], 然后C[90-120], D[60-75], E[120-130], F[75-85]
     总时间 = 130
   
   最优调度：**A先执行，总时间115分钟**

4. **Inspectors=1约束**：
   
   E和F都需要Inspectors，原来可以并行（ES(E)=60, ES(F)=75不重叠）。
   
   如果只有1个Inspector：
   - E和F必须串行
   - E在[60-70]，F必须在[70-80]
   - 但F的前提D在[60-75]完成
   - 所以F可以在[75-85]或推迟到[70-80]之后
   
   新调度：
   - 如果F在E后：E[60-70], F[70-80]，但D在[60-75]，所以F最早[75-85]
   - 实际上F的ES=max(EF(D), 前一个Inspector释放)=max(75,70)=75
   - 所以F仍为[75-85]
   
   总时间仍为85（Inspectors=2时也没有并行使用）

**考点解析**：
- CPM是项目管理的经典方法
- 关键路径决定项目最短可能时间
- 资源约束使调度问题变成NP困难
- 实际中常用启发式方法（如最小松弛优先）

### 习题11.12 【难度：⭐⭐⭐⭐】SATPlan与命题化

**知识点**：布尔可满足性、SAT规划、后继状态公理

**题目**：
考虑简单的备用轮胎问题，时间步为0和1：
- 初始：At(Spare, Trunk)⁰ = true, At(Flat, Axle)⁰ = true
- 目标：At(Spare, Axle)¹ = true

动作：
- Remove(obj, loc)：前提At(obj, loc)，效果¬At(obj, loc) ∧ At(obj, Ground)
- PutOn(t, Axle)：前提At(t, Ground) ∧ ¬At(Flat, Axle) ∧ ¬At(Spare, Axle)，效果At(t, Axle) ∧ ¬At(t, Ground)

**问题**：
1. 写出时间步0的动作命题化（列出所有可能的基元动作实例）。
2. 写出动作排除公理（同一时间不能执行两个动作）。
3. 写出前提公理（如果动作执行，前提必须为真）。
4. 写出后继状态公理（状态如何随时间变化）。

**逐步提示**：
- 动作命题化需要为每个可能的常量组合创建动作变量
- 动作排除需要列出所有动作对
- 后继状态公理使用 frame axiom 形式

**详细解答**：

1. **动作命题化（时间步0）**：
   
   ```
   Remove(Spare, Trunk)⁰
   Remove(Spare, Ground)⁰  // 初始Spare不在Ground，但理论上可能
   Remove(Flat, Axle)⁰
   Remove(Flat, Ground)⁰   // 初始Flat不在Ground
   PutOn(Spare, Axle)⁰
   PutOn(Flat, Axle)⁰      // 不合理但命题化包括
   ```
   
   实际相关动作：Remove(Spare,Trunk)⁰, Remove(Flat,Axle)⁰

2. **动作排除公理**：
   ```
   ¬(Remove(Spare,Trunk)⁰ ∧ Remove(Flat,Axle)⁰)
   ¬(Remove(Spare,Trunk)⁰ ∧ PutOn(Spare,Axle)⁰)
   ¬(Remove(Flat,Axle)⁰ ∧ PutOn(Spare,Axle)⁰)
   ...（所有动作对）
   ```
   
   实际上常用更紧凑的表示：
   ```
   Remove(Spare,Trunk)⁰ + Remove(Flat,Axle)⁰ + PutOn(Spare,Axle)⁰ ≤ 1
   ```

3. **前提公理**：
   ```
   Remove(Spare,Trunk)⁰ → At(Spare,Trunk)⁰
   Remove(Flat,Axle)⁰ → At(Flat,Axle)⁰
   PutOn(Spare,Axle)⁰ → At(Spare,Ground)⁰ ∧ ¬At(Flat,Axle)⁰ ∧ ¬At(Spare,Axle)⁰
   ```

4. **后继状态公理**：
   
   对于每个流F，公理形式：
   F¹ ↔ ActionCausesF⁰ ∨ (F⁰ ∧ ¬ActionCausesNotF⁰)
   
   ```
   At(Spare,Axle)¹ ↔ PutOn(Spare,Axle)⁰
   
   At(Spare,Ground)¹ ↔ (Remove(Spare,Trunk)⁰ ∨ At(Spare,Ground)⁰) 
                        ∧ ¬PutOn(Spare,Axle)⁰
   
   At(Spare,Trunk)¹ ↔ At(Spare,Trunk)⁰ ∧ ¬Remove(Spare,Trunk)⁰
   
   At(Flat,Axle)¹ ↔ At(Flat,Axle)⁰ ∧ ¬Remove(Flat,Axle)⁰
   
   At(Flat,Ground)¹ ↔ Remove(Flat,Axle)⁰ ∨ At(Flat,Ground)⁰
   ```

**SAT求解**：
将这些公理合取后输入SAT求解器，如果可满足，则从满足赋值中提取规划。

**考点解析**：
- SATPlan将规划问题编码为布尔可满足性问题
- 动作命题化是主要的规模爆炸来源
- 后继状态公理确保状态一致性
- 现代SAT求解器可以处理数百万变量

**10. 关于贝叶斯网络推断的复杂性，以下哪项是正确的？**

A. 所有贝叶斯网络的精确推断都是多项式时间的  
B. 单连通网络的精确推断是NP困难的  
C. 多连通网络的精确推断在最坏情况下是指数量级的  
D. 近似推断总是多项式时间的  

<details>
<summary>答案</summary>

**正确答案：C**

- A错误：多连通网络的精确推断是NP困难的
- B错误：单连通（多重树）网络可以线性时间推断
- C正确：多连通网络在最坏情况下复杂度高
- D错误：某些近似方法也可能很慢
</details>
